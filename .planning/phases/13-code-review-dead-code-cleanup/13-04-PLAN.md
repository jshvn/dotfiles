## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| new helper definition â†’ existing call-site files | A helper whose semantics drift from the inlined pattern silently changes behavior at every consolidated call site. |
| helper extraction â†’ install pipeline | Status-block helpers in particular: an incorrect helper short-circuits idempotency (the macos:shell:145 bug class re-enters). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-04-01 | Tampering | new helper changes semantics vs. original inlined pattern | mitigate | For every extraction, the executor first writes the helper to MATCH the existing inlined pattern character-for-character (modulo arg substitution); then runs `task lint && task test` AND `task --dry install` BEFORE replacing any call sites. Helper validated standalone first. |
| T-13-04-02 | Tampering | status-block helper introduces shell-var (vs. template-var) regression | mitigate | New status-block helpers use `task:` invocation form (verified viable in go-task 3.51.1, well above the 3.37 minimum). LINT-02 already catches `$VAR` in status blocks; the lint suite is the structural gate. |
| T-13-04-03 | Denial of Service | helper invocation overhead degrades `task install` runtime | accept | Helper invocation overhead in go-task is sub-millisecond per call; even 50 consolidated call sites add <100ms to `task install`. The 5-second BTSP-06 timing budget has ample headroom. |
| T-13-04-04 | Repudiation | extraction commit without call-site list | mitigate | Every extraction commit body lists every call site it consolidated (file:line). The list is auditable post-commit via `git show <SHA>`. |
</threat_model>
