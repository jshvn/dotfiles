- New helper proposal (per D-Discretion recommendation): `_:check-link-target` with vars `LINK`, `EXPECTED` — exits 0 iff `test -L "$LINK" && [[ "$(readlink -f "$LINK")" == "$EXPECTED" ]]`. Joins the `_:check-*` family.

**Existing `_:check-*` family `silent:` audit (decision recorded for the new helper):**
Inspected `taskfiles/helpers.yml` post-Phase-12 — none of the existing five helpers (`_:safe-link`, `_:check-link`, `_:check-dir`, `_:check-file`, `_:check-command`) sets `silent: true`. They all use `internal: true` + `requires.vars` + `cmds:` only. The new `_:check-link-target` helper INTENTIONALLY DIVERGES from this convention by adding `silent: true`. Justification (recorded inline in the helper body comment per the helper definition below):

- Existing `_:check-*` helpers are designed for `links:validate`-style reporting — they emit colored `check`/`cross` messages and always exit 0 (status-block-unsafe by design).
- The new `_:check-link-target` is designed for `status:`-block use — it must exit non-zero on mismatch (so go-task knows to re-run cmds:) and must NOT emit per-command output during status evaluation (status blocks should be silent to keep `task install`'s 26+ status checks from flooding stdout with helper trace lines).
- `silent: true` is the correct knob for this use case. The divergence is intentional and documented; the new helper opens its own helper-family subsection (`# Symlink Target-Match Verification`) under a section comment that calls out its different intended caller (status blocks, not validate reporting).

Notable: the `$(readlink -f ...)` form inside an inline status block is NOT a LINT-02 $VAR-in-status violation. The LINT-02 lint rule (taskfiles/lint.yml lines 158-173) excludes `$(` (command substitution) explicitly via `ggrep -vE '\$\('`. `$(readlink -f ...)` is shell command substitution, expanded by the shell executing the status command — NOT a go-task-template-time variable reference (which would be `{{.X}}`). Plan 13-05 inline path is safe under LINT-02; this is documented here so reviewers do not false-flag.

The leading-colon absolute task-reference form (e.g., `task: :identity:install` at links.yml line 131) is the established cross-namespace pattern; a helper invocation from a status block uses `task: _:check-link-target` (no leading colon needed — _: is loaded as the local helper namespace via `includes: { _: ./helpers.yml }` at links.yml line 42).
