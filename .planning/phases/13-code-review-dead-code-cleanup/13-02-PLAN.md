## Trust Boundaries

| Boundary | Description |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-13-02-01 | Tampering | source files touched by HIGH fixes | mitigate | Green-tree gate enforced after every commit: `task lint && task test` exit 0. Failures block the next commit until resolved. |
| T-13-02-02 | Denial of Service | task install pipeline regression from a HIGH fix | mitigate | After each commit that touches files in the `task install` cmd chain (`Taskfile.yml`, `taskfiles/links.yml`, `taskfiles/identity.yml`, `taskfiles/packages.yml`, `taskfiles/macos.yml`, `taskfiles/claude.yml`), run `task install` end-to-end in dry-run mode (`task --dry install`) and verify exit 0. |
| T-13-02-03 | Elevation of Privilege | security-category HIGH fixes (e.g., unsafe pipe-to-shell, missing input validation) | mitigate | Security HIGH fixes are reviewed against `docs/SECURITY.md` trust-chain documentation before commit; any net-new trust assumption is rejected (security fixes only tighten existing assumptions, never expand them). |
| T-13-02-04 | Repudiation | fix commit without REVIEW.md annotation | mitigate | Every commit's message lists the REVIEW.md row line numbers it closes; the row's `closed by` column carries the commit's short-SHA. Bidirectional traceability. |
| T-13-02-05 | Denial of Service | scope blow-out — 13-01 surfaces many more HIGH findings than planning anticipated, exceeding this plan's context budget | mitigate | Task 1 step 0 (escalation gate) counts HIGH rows + distinct touched files BEFORE any fix lands.     **Step 0 — Scope-blow-out escalation gate (FIRST action, before reading any HIGH row's body):**

    Run the exact escalation-count commands against the post-13-01 `13-REVIEW.md`:

    ```bash
    REVIEW=.planning/phases/13-code-review-dead-code-cleanup/13-REVIEW.md
    HIGH_COUNT=$(grep -cE '^\|.*\| HIGH \|' "$REVIEW")
    # Distinct file count: extract the file portion of `file:line` from each HIGH row's first column.
    # First column shape is `| <file>:<line> | HIGH | ...`. Strip leading `| ` and trailing ` |`,
    # take everything before the first `:`, dedupe.
    DISTINCT_FILES=$(grep -E '^\|.*\| HIGH \|' "$REVIEW" \
      | sed -E 's/^\|[[:space:]]*//; s/[[:space:]]*\|.*$//' \
      | awk -F: '{print $1}' \
      | sort -u \
      | wc -l \
      | tr -d ' ')
    echo "HIGH_COUNT=$HIGH_COUNT DISTINCT_FILES=$DISTINCT_FILES"
    ```

    **Escalation acceptance criterion (hard thresholds; both must hold for the plan to proceed):**

    - `HIGH_COUNT <= 15`
    - `DISTINCT_FILES <= 20`

    If EITHER threshold is breached: **STOP**. Do NOT proceed to per-file batching. Do NOT apply any fix. Instead:

    1. Print the breached threshold values to stdout.
    2. Print the recommended split shape:

Write a one-paragraph escalation note to stdout describing the file distribution (e.g., "27 HIGH rows across 24 distinct files; install/resolver.zsh has 6, taskfiles/links.yml has 4, Taskfile.yml has 3, rest are 1-2 per file.     6. Commit with message shape:
       ```
       fix(13-02): <one-line summary of the batch> — closes REVIEW.md row(s) <L1>[, <L2>, ...]
       ```
       (No body required for single-row batches; multi-row batches MAY include a body listing the rows with one-line context each.)
    7. Capture the short-SHA of the commit just made: `git rev-parse --short HEAD`.
    8. Update `13-REVIEW.md`: for each REVIEW.md row line number listed in the commit message, replace the row's `closed by` column (initially `|  |`) with `| <short-SHA> |`.     9. Stage `13-REVIEW.md` and commit with message:
       ```
       docs(13-02): annotate REVIEW.md row(s) <L1>[, <L2>, ...] closed by <short-SHA>
       ```
    10. Verify `task lint && task test` still pass after the annotation commit (markdown edit; should always pass).
    11. Proceed to next batch.

If the executor sees other related issues in the same file (a MEDIUM duplication candidate, a LOW comment-density observation, a refactor opportunity), those are NOT touched in this plan. MEDIUM/LOW rows belong to Plan 13-06; refactor opportunities belong to Plan 13-04 (duplication) or Plan 13-03 (dead code). Phase scope discipline.

Time-box: if after reading the file and the surrounding context the executor still cannot derive a fix, STOP, annotate the REVIEW.md row's `closed by` column with `defer: needs-investigation — Plan 13-06`, and proceed to the next batch. The downstream Plan 13-06 will pick up the investigation as a re-classified MEDIUM. Do NOT fabricate a fix.

Until the structural gate is green, the rest of the green-tree gate cannot enforce. After the structural fix lands, the standard batch loop runs.
