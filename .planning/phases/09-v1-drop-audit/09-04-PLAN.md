<objective>
Produce `shards/docs.md` — the audit rows for every substantive v1-only doc fragment compared against the v2 `docs/` set (AUDIT-05). Primary input is `install/README.md`; secondary inputs are any `.md` files inside `install/` that are not part of the v2 doc surface; tertiary input is git-history archeology of doc fragments under a prior top-level README.md (treat as optional — out of scope unless trivial per 09-CONTEXT.md `<deferred>`).

Purpose: AUDIT-05's success criterion (ROADMAP Phase 9 SC#4) is that every v1-only doc fragment substantive enough to be useful is logged for Phase 10 / Phase 14 disposition. The v2 docs/ set (`docs/MANIFEST.md`, `docs/SECURITY.md`, `docs/CUTOVER.md`, `docs/MIGRATION.md`, `docs/MACHINES.md`, `docs/README.md`) is the comparison baseline. A v1 fragment whose content is already paraphrased in v2 is classified `ported / drop`; a fragment with unique operator information is classified `keep` with a named v2 owner; a fragment that is purely v1-historical (now-obsolete profile-model commentary, etc.) is classified `dropped / drop`.

Output: One shard file with rows for every classified v1 doc fragment, ready for plan 09-05 to concatenate verbatim under AUDIT.md's `## Docs` section.
</objective>

    **Step 1 — Inventory.**
    Enumerate every section in `install/README.md` via `grep -n '^##' install/README.md`.     **Step 2 — Per-fragment classification.**
    For each (path, heading, line_range) tuple from Step 1, run a grep comparison against the six v2 docs files to decide v2 status:

    ```
    grep -nFi -- '<key phrase from v1 heading or paragraph>' \
      docs/MANIFEST.md docs/SECURITY.md docs/CUTOVER.md \
      docs/MIGRATION.md docs/MACHINES.md docs/README.md
    ```

Multiple greps per section are fine — record the literal grep commands as the per-row rationale evidence so a reviewer can re-run them.

    Classify per D-09 (operator-visible information equivalence, not byte-equivalence):
    - **ported** — v2 docs/ carries the same operator-visible information (paraphrased OK).
    - **partially-ported** — v2 docs/ covers most of the topic but misses one or more sub-points.
    - **dropped** — no v2 docs equivalent exists.

    Decide keep/drop:
    - **drop** when v2 status is ported and the v1 fragment adds nothing the v2 docs don't already carry, OR when the fragment is v1-historical commentary the v2 model retired (profile-suffix Brewfiles, hostname inference, etc.) — these are intentionally retired per D-09.
    - **keep** when v2 status is partially-ported (must port the missing sub-point) OR dropped (must port the entire fragment) AND the content has Phase-10/14 operator value.

    v2 owner column for keep rows:
    - manifest-model content → `docs/MANIFEST.md`
    - bootstrap/install procedure / trust chain → `docs/SECURITY.md` (trust chain) or `install/README.md` itself (procedure prose can stay in install/README.md after Phase 14 trim)
    - cutover-specific → `docs/CUTOVER.md` (transient — Phase 11 RMV-04 retires this file; Phase 14 TRIM-03 collapses any kept fragments into `docs/MIGRATION.md` or removes them)
    - migration history / v1-to-v2 mapping → `docs/MIGRATION.md`
    - per-machine info → `docs/MACHINES.md`
    - top-level pointer → `README.md`
    - Phase 5 Brewfile transitional commentary → `packages/README.md` (already created in Phase 5)
    - install-engine internals (`resolver.zsh`, `messages.zsh`, `cutover-gate.zsh` callout sections) → `install/README.md` itself stays the owner if the content is install-internals documentation (Phase 14 may trim, but the owner doesn't change)
