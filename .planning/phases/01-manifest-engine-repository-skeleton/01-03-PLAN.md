       e. delete manifests/machines/_addmachine-test.toml

Fixture-test merge pipeline (inline yq+jq for manifest:test, per Plan 03 IMPLEMENTATION NOTE option (a)):
  Pass 1: merged=$(yq eval-all '. as $i ireduce ({}; . * $i)' "$fix/defaults.toml" "$fix/machine.toml" -o json)
  Pass 2: extras_def=$(yq -o=json '.packages.brew.extra_packages // []' "$fix/defaults.toml"); extras_mach=$(yq -o=json '.packages.brew.extra_packages // []' "$fix/machine.toml"); union=$(echo "$extras_def $extras_mach" | jq -s 'add | unique')
  Final: actual=$(echo "$merged" | jq --argjson extras "$union" '.packages.brew.extra_packages = $extras')
  Compare: diff <(echo "$actual" | jq -S .) <(jq -S . "$fix/expected.json")

NOTE on fixture-06 extra_packages: Plan 01's fixture-06 expected.json should match the output of `jq -s 'add | unique'` (which sorts alphabetically — `["docker-desktop","jq","yq"]`). If Plan 01 chose a different ordering, this fixture-test pipeline must match that ordering exactly. Read Plan 01's actual fixture-06 expected.json to confirm before implementing the pipeline. If the orderings disagree, prefer adjusting THIS pipeline to match Plan 01 (since Plan 01's expected.json is the spec).

NOTE on fixtures 01-05: they don't have packages.brew.extra_packages keys, so the union step yields `[]` and the final assignment is a no-op for those fixtures (the merged output already lacks the key, and assigning `[]` would ADD the key incorrectly). The pipeline must skip the union/assign step when neither side declares extra_packages — check via `yq 'has("packages") and (.packages | has("brew")) and (.packages.brew | has("extra_packages"))'` on each input file; only run the union step if either returns true.
