    **Function 1 -- `apply_shell_registration()`:**
    - First branch: `if ! grep -qxF "$BREW_ZSH" /etc/shells; then info "Adding Homebrew zsh to /etc/shells..."; echo "$BREW_ZSH" | sudo tee -a /etc/shells > /dev/null; success "Homebrew zsh added to /etc/shells"; fi`. The `info` line printed BEFORE the sudo invocation makes the prompt explicable (RESEARCH Pitfall 7). The `> /dev/null` suppresses tee's stdout (it would otherwise duplicate the path to the terminal). The `-qxF` flag triple is whole-line + literal-string + quiet (RESEARCH Don't-Hand-Roll).
    - Second branch: `local current_shell; current_shell=$(dscl . -read /Users/$USER UserShell 2>/dev/null | awk '{print $2}'); if [[ "$current_shell" != "$BREW_ZSH" ]]; then info "Changing default shell from $current_shell to $BREW_ZSH..."; chsh -s "$BREW_ZSH" || { error "chsh failed; run manually: chsh -s $BREW_ZSH"; exit 1; }; fi`. `dscl . -read /Users/$USER UserShell` is the canonical read of the REGISTERED login shell, NOT `$SHELL` (RESEARCH Don't-Hand-Roll: `$SHELL` reflects the running shell process). The `awk '{print $2}'` extracts the path from `UserShell: <path>` output (single line on macOS 14+ per RESEARCH Pitfall 4; the `head -n 1` defense is cheap insurance against future variance -- recommend adding it: `dscl . -read /Users/$USER UserShell 2>/dev/null | head -n 1 | awk '{print $2}'`). The `chsh` failure error message tells the user the exact manual recovery command.

    Order matters (RESEARCH Pitfall 8): `/etc/shells` append MUST happen before `chsh`, because macOS chsh validates the target against `/etc/shells` and refuses non-standard shells.
