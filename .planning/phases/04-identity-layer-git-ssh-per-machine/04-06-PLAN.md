<!-- The working pattern (one-hop) and the broken pattern (two-hop) side-by-side. -->

Working pattern (taskfiles/identity.yml:67):
  RESOLVED_JSON_PATH: '{{.XDG_STATE_HOME}}/dotfiles/resolved.json'

This substitutes correctly under both standalone invocation AND the
include-with-vars-override path used by Taskfile.yml. The single template
hop refers ONLY to the parent-forwarded XDG_STATE_HOME var, so go-task's
include-vars evaluation sees the variable in scope.

Broken pattern (taskfiles/manifest.yml:51-53):
  STATE_DIR: '{{.XDG_STATE_HOME}}/dotfiles'
  STATE_FILE: '{{.STATE_DIR}}/machine'
  RESOLVED_JSON_PATH: '{{.STATE_DIR}}/resolved.json'

When manifest.yml is loaded as an include, go-task renders the
MANIFEST_JSON sh: text against a scope where the parent-forwarded
XDG_STATE_HOME IS available but the locally-derived intermediate
STATE_DIR is NOT in scope. The chain collapses to '/resolved.json'.
go-task issues #1295, #2108, #535 collectively document this behaviour.

Why STATE_DIR survives in cmds: blocks (and therefore stays in the file):
  cmds: blocks render their template substitutions LATER in the lifecycle
  -- after vars-block sh: eval has completed and all template vars are
  populated. So `mkdir -p "{{.STATE_DIR}}"` inside cmds: substitutes
  correctly even though `RESOLVED_JSON_PATH: '{{.STATE_DIR}}/resolved.json'`
  inside the vars: block does not. Three cmds: callers depend on STATE_DIR
  today (lines 135, 158, 540 -- mkdir setup invocations); they continue
  to work without modification.

STATE_FILE (line 52) -- also derived from STATE_DIR -- is referenced ONLY
by sh: vars at line 69 and by cmds: blocks at lines 130, 136, 152, 205,
247, 369, 411, 487, 506, 508, 511, 513, 541. The line-69 sh: var
(`MACHINE: sh: head -n1 '{{.STATE_FILE}}' ...`) hits the same
include-vars-evaluation issue as RESOLVED_JSON_PATH but is BENIGN: when
STATE_FILE collapses to `/machine`, head fails silently and MACHINE
becomes empty, which is the intended fallback path (downstream code
handles empty MACHINE explicitly via the AVAILABLE_MACHINES error
message). However, leaving STATE_FILE on the two-hop chain is
inconsistent with the fix shape and could introduce future bugs if a
new sh: var is added that consumes STATE_FILE. SCOPE NOTE: this plan
fixes ONLY RESOLVED_JSON_PATH because that is the documented gap-2
truth; STATE_FILE is a follow-up consideration recorded in the SUMMARY.
