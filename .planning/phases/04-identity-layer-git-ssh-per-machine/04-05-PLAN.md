go-task include semantics (verified on go-task 3.50.0; project minimum 3.37):
- Tasks declared in a file included as `<alias>:` are exposed as `<alias>:<key>`.
  So `taskfiles/manifest.yml` included under alias `manifest:` with a bare key
  `resolve:` produces the externally visible name `manifest:resolve`.
- Intra-file `task:` references inside a `cmds:` list resolve against the
  LOCAL key first, BEFORE the include alias is prepended. So inside
  taskfiles/manifest.yml, `- task: resolve` correctly invokes the file's local
  `resolve:` task even though externally it is named `manifest:resolve`.
- Cross-namespace `deps:` references from inside an included taskfile are
  resolved RELATIVE to the current include namespace unless they begin with a
  leading colon. From inside `identity:`, `deps: [manifest:resolve]` becomes
  `identity:manifest:resolve` (broken). The leading-colon form
  `deps: [:manifest:resolve]` anchors at the root namespace and resolves
  correctly. The leading-colon syntax is documented in go-task and present
  since well before 3.37.
- Root-level `deps:` references in the root Taskfile.yml DO see the included
  namespaces directly (no enclosing namespace), so `deps: [manifest:resolve]`
  works without a leading colon.
