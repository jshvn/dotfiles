<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repo author -> `packages/*.rb` content | Bundle file contents executed by `brew bundle install` (Plan 04); a malicious bundle line is direct package installation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | T (Tampering) | `packages/core.rb` + `packages/gui.rb` | accept | Bundles are checked into the repo and reviewed at PR time. No runtime input. The composer (Plan 03) concatenates verbatim with no eval -- a malicious line would have to ship through git review. Standard repo-supply-chain trust boundary; documented in `docs/SECURITY.md` (Phase 2). |
| T-05-02 | I (Information disclosure) | bundle files | accept | Bundle contents are formula/cask names only (public Homebrew identifiers). No secrets, no credentials, no PII. |
| T-05-03 | D (DoS) | malformed `# verify:` comment | mitigate | Verify-parser regex in Plan 04 is anchored (`^[[:space:]]*brew ...` / `^[[:space:]]*cask ...`) and tolerates any content past `# verify: ` up to end-of-line; a malformed comment produces a per-package verify failure (printed in the check/cross table per D-07), not a parser crash. |
| T-05-04 | S (Spoofing) | typo in `# verify: <App>` for a cask | accept | A typo (`# verify: 1Pasword` for the 1password cask) produces a false-positive verify failure after install. D-04 trades doubled-authorship-cost for verify reliability. The typo is a build-time error caught on the FIRST run of `task packages:verify`. |

Block on HIGH; this plan has none. T-05-03 mitigation is implemented in Plan 04; this plan only authors the source content with the comment shape Plan 04 expects.
</threat_model>
