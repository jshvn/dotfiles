## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Non-interactive zsh contexts (cron, scp, scripts) → shell/.zshenv | .zshenv is sourced by EVERY zsh invocation, including non-interactive ones. Any crash here breaks cron jobs, scp uploads, and remote scripts. |
| $XDG_STATE_HOME/dotfiles/{machine,resolved.json} → shell startup | The state files are written by `task setup` and `task manifest:resolve`. They are repo-local data, not network-sourced. |
| Login shell → SSH_AUTH_SOCK assignment | .zprofile sets SSH_AUTH_SOCK based on a feature flag read from resolved.json. A malformed resolved.json or a tampered features.one-password-ssh value could redirect SSH agent socket lookup. |
| antidote `bundle` command → github.com plugin fetch | First-shell-start triggers `antidote bundle` which clones git repos. This is the same trust posture as v1's antigen — accepted (T-03-02 from Plan 01). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Denial of Service | shell/.zshenv missing-state crash | mitigate | .zshenv guards the state-file read with `[[ -r "${XDG_STATE_HOME}/dotfiles/machine" ]]`. On missing file, DOTFILES_MACHINE stays unset (no crash). cron/scp invocations work without `task setup`. |
| T-03-05 | Tampering | resolved.json `features."one-password-ssh"` value | accept | resolved.json is written by `install/resolver.zsh` from repo-committed TOMLs. Tampering requires file write to $XDG_STATE_HOME/dotfiles — a local-only attack surface. Worst case is the user's SSH_AUTH_SOCK pointing at a wrong socket; ssh fails noisily and the user fixes the manifest. |
| T-03-06 | Information Disclosure | shell/.zshrc missing-machine warning to stderr | accept | Warning emits the literal string "no active machine selected" to stderr in interactive shells. No PII, no machine identifiers, no path leaks. |
