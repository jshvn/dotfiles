## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `manifests/*.toml` → `resolved.json` | Repository-controlled TOML is read by `install/resolver.zsh` (Phase 1) and emitted as JSON. Threat surface is bounded — repo files are version-controlled and reviewed. |
| `configs/antidote/zsh_plugins.txt` → antidote runtime | Plain-text bundle list is read by `antidote bundle` at shell startup (Plan 02 wires this). Antidote fetches plugins over HTTPS from github.com on first invocation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `configs/antidote/zsh_plugins.txt` | accept | Plugin list is committed to the repo; tampering requires repo write access. Antidote pins to git refs only as supplied; we accept default branch tracking for ohmyzsh/zsh-users (the same trust posture as v1's antigen). No PII or secrets in the file. |
| T-03-02 | Spoofing | github.com plugin sources | accept | Bundle list references `ohmyzsh/ohmyzsh` and `zsh-users/*` — well-known maintained orgs. HTTPS+TLS is antidote's transport; we trust the same chain as v1. Future hardening (v2): pin to commit SHAs. |
| T-03-03 | Information Disclosure | Manifest feature flags | accept | Feature flags (`ghostty`, `jgrid-net`, `macos-finder`) describe local install capabilities only — no secrets, no machine identifiers, no IP addresses. Public commit history is fine. |
</threat_model>
