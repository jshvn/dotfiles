       - Narrative: v1 used `Brewfile-<profile>.rb`, `zsh/aliases/<profile>/`, and hostname-based switches to select state. v2 replaces this with a single TOML per machine under `manifests/machines/<name>.toml` that inherits from `manifests/defaults.toml`.        - Path table example rows: `Brewfile-personal.rb -> packages/core.rb + extra_packages in manifests/machines/personal-laptop.toml`; `zsh/aliases/server.zsh -> shell/aliases/<topic>.zsh (flat) gated by feature flag in manifests/machines/server-1.toml`.

    2. `## Antigen -> Antidote`
       - Narrative: Antigen (archived January 2018) replaced by Antidote (static plugin manifest, faster startup, primary lever for the 200ms cold-start target per SHEL-04). v1 prompt (alanpeabody-based theme) ported as-is to `shell/theme.zsh` — Starship was rejected as a v1 change.
       - Path table rows: `zsh/.antigenrc -> configs/antidote/zsh_plugins.txt`; `zsh/theme.zsh -> shell/theme.zsh (ported as-is)`.

    3. `## Brewfile-<profile>.rb -> packages/<purpose>.rb + extra_packages`
       - Narrative: v1 had per-profile Brewfiles. v2 has per-purpose Brewfiles (`core.rb`, `gui.rb`) that machine TOMLs list in `[packages.brew].bundles`, plus per-machine `extra_packages` typed sub-tables (`formulae`, `casks`, `mas`) for one-offs. Cask isolation in `gui.rb` lets server machines decline GUI bundles via manifest (PKGS-05).
       - Path table rows: `install/Brewfile-personal.rb -> packages/core.rb + packages/gui.rb + manifests/machines/personal-laptop.toml extra_packages`; `install/Brewfile-server.rb -> packages/core.rb only (gui bundle declined via bundles list)`.

    4. `## zsh/ -> shell/ (flat layout)`
       - Narrative: v1 nested aliases/functions by platform subdirectory. v2 flattens because all four target machines are macOS in v1 (Linux deferred to v2+); one alias per file, one function per file. Future Linux re-introduction is acknowledged as a known cost in PROJECT.md.
       - Path table rows: `zsh/aliases/darwin/<topic>.zsh -> shell/aliases/<topic>.zsh`; `zsh/functions/<name>.zsh -> shell/functions/<name>.zsh`; `zsh/.zshrc -> shell/.zshrc`.

    5. `## gsd-install -> sentinel-gated claude:gsd + explicit claude:update`
       - Narrative: v1's `gsd-install` ran `npx` on every `task install` (expensive, network-required). v2 uses a presence sentinel under `$XDG_STATE_HOME/dotfiles/` as the `status:` check for `claude:gsd`; `npx` runs only when the sentinel is absent. An explicit `task claude:update` deletes the sentinel and re-runs. Cites CLDE-03.
       - Path table rows: `gsd-install task -> task claude:gsd (sentinel-gated)`; `(no v1 equivalent for forced update) -> task claude:update`.

    6. `## Hostname-based Match exec -> manifest-driven identity gates`
       - Narrative: v1's SSH config used `Match exec` against `hostname` (the `.zprofile:55-56` bug class — burned us before). v2 reads the machine name from `$XDG_STATE_HOME/dotfiles/machine` (set by `task setup`) and selects identity via manifest `[identity]` block; `taskfiles/identity.yml` materializes the `Include` directives via `_:safe-link`. No `hostname` references anywhere in the identity path (IDNT-01..IDNT-08).
       - Path table rows: `ssh/configs/*-Match-exec-hostname -> identity/ssh/identities/<identity>/`; `git config user.email switch by hostname -> includeIf gitdir in identity/git/identities/<identity>/config`.

    7. `## macos:shell $BREW_ZSH -> {{.BREW_ZSH}}`
       - Narrative: v1 `macos:shell:145` bug used `$BREW_ZSH` shell variable inside a `status:` block where it expanded to empty (shell vars are not in scope during status evaluation). This caused the task to re-run on every install. v2 uses `{{.BREW_ZSH}}` template var resolved at task-graph build time. LINT-02 enforces this structurally for every taskfile.
       - Path table rows: `macos.yml status: $BREW_ZSH -> macos.yml status: {{.BREW_ZSH}}`.

    Final two sections:

    `## Rollback` H2 — operator procedure for falling back to v1 if v2 regresses on machine X during the cutover window:
    - Step 1: stop using v2 on machine X (close v2 shells).
    - Step 2: revert the active machine sentinel: `rm $XDG_STATE_HOME/dotfiles/machine` and `rm $XDG_STATE_HOME/dotfiles/cutover-ack` (un-cuts the machine).
    - Step 3: re-source v1 zsh files from the v1 repo (which is still installed during cutover window per CUTV-06 archive-not-delete).
    - Step 4: open an issue / note the regression in the per-machine notes column of `docs/CUTOVER.md`; status back to `planning`.

    `## Archiving v1` H2 — operator procedure for archiving v1 after the LAST machine cuts over:
    - Pre-condition: all four rows in `docs/CUTOVER.md` § "Per-machine cutover state" show status `cut-over` AND days-on-v2 >= 7.
    - Step 1: rename the v1 directory (do not delete — CUTV-06): `mv ~/Git/personal/dotfiles-v1 ~/Git/personal/dotfiles-v1.archive`.
    - Step 2: if v1 is tracked by a separate remote/branch, archive that branch via `git push origin master:archive/v1`.
    - Step 3: update docs/CUTOVER.md per-machine state column: status -> `archived`.

    No emojis. No AI attribution. Inline-code for every command, file path, task name. Heading depth: H1 + H2 (H3 only inside Rollback/Archiving step lists if absolutely required).
