<objective>
Add a top-level `validate:` task to the root `Taskfile.yml` that composes every per-component validate (manifest, identity, links, macos, packages, claude) into a run-all-aggregate diagnostic per D-04 and D-05. Each component runs unconditionally regardless of prior failures (via `ignore_error: true` on each `task:` delegation), then a final shell-block re-evaluates each component's exit code by re-invoking the validate (idempotent read-only cost is negligible per 08-RESEARCH.md Pitfall 5) to compute a check/cross summary table and bubble up the aggregate exit code.

Feature-off components (e.g., `claude:validate` on server-1 with `claude-marketplace = false`) appear in the summary as `n/a` per D-06; the per-component validate emits a sentinel info line that the aggregator's summary block greps for to distinguish skip-from-feature-off vs pass-from-clean-state.

Purpose: CUTV-01 — a single composed `task validate` is the cutover-day signal that a machine is in the expected state. Per D-07 it stays runtime-only (Tier 1); `task lint` (Tier 0) and `task test` (Tier 3) remain separate top-level entries. This plan ships the aggregator; per-component validates were shipped in their respective phases (Phase 1 manifest, Phase 4 identity, Phase 5 packages, Phase 6 macos, Phase 7 claude, Phase 8 P1 links).

Output: A modified `Taskfile.yml` (~35 added lines) — no new file, no new include. The aggregator lives at root because every component validate is already included; adding a wrapper module would require an extra includes: entry for a one-task file.
</objective>

<interfaces>
Composed validate consumes six per-component validates (already shipped). Their behavior contract:

| Component | Task | Exit on failure | Feature gate |
|-----------|------|-----------------|--------------|
| manifest  | `:manifest:validate` | non-zero (delegates to resolver --validate-only) | always-runs |
| identity  | `:identity:validate` | composed: validate:symlinks (0), validate:git (non-zero), validate:ssh-add (gated), validate:keys (non-zero) — confirmed in 08-RESEARCH.md "Existing Code Survey" + Assumptions Log A3 | always-runs at aggregator level; sub-tasks self-gate on identity features |
| links     | `:links:validate` (FIXED in P1) | non-zero on any failed symlink (failures counter + `exit "$failures"`) | always-runs at task level; per-target feature gates via inline ternaries in EXPECTED_TARGETS |
| macos     | `:macos:validate` | non-zero (`exit "$failed"`); enumerate-all pattern at taskfiles/macos.yml lines 254-305 | per-concern feature gates inside shell block |
| packages  | `:packages:validate` | non-zero (wraps `packages:verify` which exits failures count) | always-runs |
| claude    | `:claude:validate` | non-zero on missing CLI / marketplace / plugin / GSD sentinel; exits 0 early when `claude` CLI missing (taskfiles/claude.yml line 240) — per Pitfall 5 the aggregator's summary cannot rely solely on exit code for claude when CLI missing; not a blocker for v1 because all 4 target machines have claude installed | always-runs at aggregator level; gated internally on `claude-marketplace` |

D-06 feature-off skip protocol: per-component validate emits the sentinel info line `feature disabled -- skipped` to stdout when the component's primary feature flag is off; the aggregator greps stdout for that exact substring to render the summary row as `n/a`.

For Phase 8 v1 the only per-component validates that actually feature-gate at the top level are: `claude:validate` (gated on `claude-marketplace`) and parts of `identity:validate` (e.g., `validate:ssh-add` gated on `one-password-ssh`). The implementation under this plan adopts D-06 sentinel-emission for `claude:validate` only as a minimal change (1-line addition to claude:validate's early-exit branch); other per-component validates already self-skip internally without needing aggregator awareness.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user shell → `task validate` | command-line invocation; no untrusted input |
| component validate output → summary grep for sentinel | sentinel substring is a fixed string under our control; not user input; grep is bounded to a known fixed pattern |
| six per-component validate re-invocations | each is an idempotent read-only check on local state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-04 | Tampering | D-06 sentinel substring matching | mitigate | sentinel is exact-match fixed substring under repo control (`feature disabled -- skipped`); validate emitters in this repo are the only sources |
| T-08-05 | Denial of service | double-invocation of six per-component validates | accept | per 08-RESEARCH.md A1+Pitfall 5: each validate is idempotent read-only; total double-run cost measured in milliseconds on personal-laptop; well under LINT-08 5s gate |
| T-08-06 | Repudiation | summary exit code does not reflect actual component state | mitigate | summary block re-invokes each component independently and captures `$?`; the `ignore_error: true` per-entry never reaches the summary because the summary uses its own subshell with `$rc` |
| T-08-SC | Tampering | npm/pip/cargo installs | n/a | This plan installs no packages — pure Taskfile.yml addition |
</threat_model>
