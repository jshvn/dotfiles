<objective>
Add a top-level `validate:` task to the root `Taskfile.yml` that composes every per-component validate (manifest, identity, links, macos, packages, claude) into a run-all-aggregate diagnostic per D-04 and D-05. Each component runs unconditionally regardless of prior failures, then a final shell-block re-evaluates each component's exit code by re-invoking the validate (idempotent read-only cost is negligible per 08-RESEARCH.md Pitfall 5) to compute a check/cross summary table and bubble up the aggregate exit code.

Before writing the aggregator, this plan empirically resolves the open question flagged as 08-RESEARCH.md assumption A1: does go-task 3.37 support per-entry `ignore_error: true` on `task:` invocations inside a `cmds:` list? The probe task (Task 1) writes a minimal `/tmp/probe.yml` with two `task:` entries (one intentionally failing) plus `ignore_error: true`, runs it, and confirms whether the second entry still executes. The aggregator implementation in Task 3 uses whichever mechanism the probe validates as working: per-entry `ignore_error: true` if supported, or the `task X || true` shell-fallback pattern otherwise.

Feature-off components (e.g., `claude:validate` on server-1 with `claude-marketplace = false`) appear in the summary as `n/a` per D-06; the per-component validate emits a sentinel info line that the aggregator's summary block greps for to distinguish skip-from-feature-off vs pass-from-clean-state.

Purpose: CUTV-01 — a single composed `task validate` is the cutover-day signal that a machine is in the expected state. Per D-07 it stays runtime-only (Tier 1); `task lint` (Tier 0) and `task test` (Tier 3) remain separate top-level entries. This plan ships the aggregator; per-component validates were shipped in their respective phases (Phase 1 manifest, Phase 4 identity, Phase 5 packages, Phase 6 macos, Phase 7 claude, Phase 8 P1 links).

Output: A modified `Taskfile.yml` (~35 added lines) + a 1-line addition to `taskfiles/claude.yml` (D-06 sentinel). No new file, no new include. The aggregator lives at root because every component validate is already included; adding a wrapper module would require an extra includes: entry for a one-task file.
</objective>
