## What Was Done

### Task 1 — Move HISTFILE block

**Problem:** `.zshenv` set `HISTFILE="$XDG_DATA_HOME/zsh/history"` with a comment claiming this was
the correct location to run before VS Code shell-integration. This was exactly inverted from reality:

- Apple's `/etc/zshrc` runs **before** the user's `.zshrc` (and before `.zshenv` has any effect on
  interactive-shell history I/O) and unconditionally sets `HISTFILE=${ZDOTDIR:-$HOME}/.zsh_history`.
- VS Code's `shellIntegration-rc.zsh` sets `HISTFILE=$USER_ZDOTDIR/.zsh_history` before sourcing
  the rest of the user's `.zshrc`.

Both clobbered the `.zshenv` assignment. `.zshrc` is the only init file that runs **after** both,
so an assignment at the top of `.zshrc` is the last write and wins.

**Changes to `shell/.zshrc`:**
- Inserted HISTFILE block immediately after the header banner (line 20-32), before the ZSH_COMPDUMP
  section (line 36).
- Block contains: `export HISTFILE`, `export HIST_STAMPS`, `export HISTSIZE=50000`,
  `export SAVEHIST=50000`, `mkdir -p "${HISTFILE%/*}"`, `setopt SHARE_HISTORY`.
- Added explanatory comment naming both clobbering sources by name.
- Updated `# Side effects:` banner to list HISTFILE, HIST_STAMPS, HISTSIZE, SAVEHIST,
  SHARE_HISTORY setopt, and HISTFILE parent dir creation.

**Changes to `shell/.zshenv`:**
- Removed the 6-line HISTFILE block (comment + 4 exports/commands + setopt).
- Collapsed to a clean single blank line between ZDOTDIR and the CLAUDE_CONFIG_DIR comment.
- Updated `# Side effects:` banner to remove HISTFILE, HIST_STAMPS, creates-parent-dir, and
  enables-SHARE_HISTORY clauses.

**Verification:**
- `zsh -n shell/.zshenv` passes.
- `zsh -n shell/.zshrc` passes.
- HISTFILE block appears at line 27, ZSH_COMPDUMP at line 36 (ordering correct).
- `.zshenv` contains no HISTFILE/HIST_STAMPS/SHARE_HISTORY references.

## Follow-Up Correction (commit 70de621)

The initial fix landed the HISTFILE block at the **top** of `.zshrc`. User repro
showed the bug was not fixed: `~/.zsh_history` was still created when typing a
command in a fresh VS Code terminal.

**Missed analysis:** [shell/.zshrc:105-110](../../../../shell/.zshrc#L105) manually
sources VS Code's `shellIntegration-rc.zsh` — that source runs **after** the
top-of-zshrc HISTFILE assignment and overwrites it via:

```sh
if [[ "$VSCODE_INJECTION" == "1" ]]; then
    HISTFILE=$USER_ZDOTDIR/.zsh_history
fi
```

When VS Code is launched from the Dock/launchd with no inherited `ZDOTDIR`,
`USER_ZDOTDIR` is captured as `$HOME`, so `HISTFILE` lands at `$HOME/.zsh_history`.

**Repro under simulated VS Code env:**
```
$ USER_ZDOTDIR=$HOME VSCODE_INJECTION=1 TERM_PROGRAM=vscode zsh -i -c 'echo $HISTFILE'
/Users/josh/.zsh_history      # BUG
```

**Fix:** Moved the HISTFILE block from the top of `.zshrc` (above ZSH_COMPDUMP) to
**below** the VS Code integration source block. New comment names the three
clobbering sources (1) Apple's `/etc/zshrc`, (2) VS Code's `shellIntegration-rc.zsh`,
(3) antigen/OMZ `history.zsh`. Placement makes the assignment the last write.

**Verification under simulated VS Code env (commit 70de621):**
```
$ USER_ZDOTDIR=$HOME VSCODE_INJECTION=1 TERM_PROGRAM=vscode zsh -i -c 'echo $HISTFILE'
/Users/josh/.local/share/zsh/history   # FIXED
```

**Lessons:**
- "Top of `.zshrc`" was not the last write — the project's own VS Code integration
  source clobbered it.
- The verify gate `zsh -i -c 'echo $HISTFILE'` passed in this Claude Code shell
  because Claude Code's parent process had `ZDOTDIR=$HOME/.config/zsh` already set
  in env, so USER_ZDOTDIR was correct and VS Code's integration assigned
  `HISTFILE=$ZDOTDIR/.zsh_history` (not `$HOME/.zsh_history`). That assignment is
  still wrong but didn't surface during verification because the file path happened
  to differ from what the user originally reported. The fix verification must
  simulate the actual launchd-spawned VS Code env: `USER_ZDOTDIR=$HOME`.
