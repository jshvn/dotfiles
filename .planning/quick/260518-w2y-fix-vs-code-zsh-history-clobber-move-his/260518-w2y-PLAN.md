Operative ordering facts:
- Apple's /etc/zshrc sets `HISTFILE=${ZDOTDIR:-$HOME}/.zsh_history` and runs BEFORE the user's .zshrc.
- VS Code's shell-integration script (/Applications/Visual Studio Code.app/.../shellIntegration-rc.zsh:22) sets `HISTFILE=$USER_ZDOTDIR/.zsh_history` and is sourced FROM the user .zshrc near the top, then sources the rest of the user .zshrc afterwards. Setting HISTFILE before that source would still be overwritten unless we set it AFTER. However, the integration script is sourced conditionally near line 89 of .zshrc, well after our new top-of-file block — but it then loads the rest of .zshrc; this means: place HISTFILE BEFORE the VS Code integration source so it gets clobbered, then... wait — that's wrong. Re-read: VS Code's wrapper renames the user's .zshrc to USER_ZDOTDIR and sources it itself; ordering depends on VSCODE_SHELL_INTEGRATION env var, not on our .zshrc internal order. The reliable fix is: set HISTFILE at the TOP of .zshrc; /etc/zshrc has already run; VS Code's integration set HISTFILE before sourcing our .zshrc; our assignment at the top WINS. This matches the orchestrator's analysis.
